# AGENTS.md

## Mission

Build `lci_reduce`, a ZIP-only Python tool that creates a lite openLCA JSON-LD database.

The core workflow removes only non-selected elementary exchanges from process JSON files.

The mathematical reduction must be correct and the scope of every guarantee must be stated precisely.

## No IPC

Do not use openLCA IPC.

Do not connect to openLCA.

Do not edit an active openLCA database.

Primary input is JSON-LD ZIP. Primary output is JSON-LD ZIP.

Read-only sidecar workflows may emit analysis files without creating a lite database.

## Main workflow

Input:
- original openLCA JSON-LD database ZIP;
- optional LCIA methods JSON-LD ZIP or folder;
- tau value;
- LCIA indicator selection.

Output:
- lite JSON-LD database ZIP;
- exchange manifest;
- process manifest;
- validation JSON;
- config JSON;
- warnings CSV;
- short PDF report.

## Strict database preservation rules

Never remove or modify:
- technosphere exchanges;
- product/reference exchanges;
- waste exchanges;
- provider links;
- allocation data;
- process parameters;
- uncertainty metadata on retained exchanges;
- flow objects;
- unit objects;
- category objects;
- actor objects;
- source objects;
- LCIA method objects;
- product system objects.

Only unselected elementary exchange objects may be removed from process exchange lists.

Do not regenerate UUIDs.

Do not merge flows.

Do not infer missing links silently.

Do not silently drop anything.

Unused flow objects may remain in the lite database.

## Mathematical rule

For each process, build:

A[j, e] = exchange_amount(e) × characterisation_factor[j, flow(e)]

Rows are selected LCIA categories.

Columns are elementary exchange objects.

Reduction uses characterised contributions, not raw exchange amounts.

This is an LCIA-conditioned reduction for the selected indicators only.

Split signed contributions:

A_pos = max(A, 0)

A_neg = max(-A, 0)

Run deterministic greedy tau-cover separately on A_pos and A_neg.

Retain the union:

selected = selected_pos OR selected_neg OR protected

## Coverage requirement

For every active positive category:

retained_pos[j] >= tau × full_pos[j]

For every active negative category:

retained_neg[j] >= tau × full_neg[j]

If coverage verification fails, abort.

## Greedy algorithm

At each step, choose the unselected exchange with the largest weighted marginal fill:

score[e] = sum_j min(contribution[j, e], remaining_demand[j]) / full[j]

This weighting makes the greedy choice invariant to simple rescaling of category magnitudes.

Break ties deterministically using stable exchange ID, then original order.

The greedy algorithm is deterministic but not globally optimal.

Do not claim global optimality.

Do not describe the implemented rule as raw marginal fill.

## Default policies

Uncharacterised elementary exchanges are kept by default.

Unit handling is strict by default.

If unit compatibility is ambiguous, fail.

If characterization is unavailable for an elementary exchange, make the handling policy explicit.

## Do not implement

Do not implement:
- openLCA IPC;
- speed-up analysis;
- reducibility curves;
- reducibility index;
- plotting;
- benchmarking dashboards;
- paper tables.

## Tests

Mathematical tests are mandatory.

Run pytest before considering the task complete.

The task is not complete if any coverage test fails.


## Additional workflow: LCIA-critical flow priority file

In addition to tau-reduced database creation, the tool may generate an LCIA-critical flow priority file for LCI database transfer and mapping repair.

This workflow is a sidecar-generation workflow only.

It must not rewrite the database.

It must not create a lite database ZIP.

It must not remove exchanges.

It must not modify process JSON files.

Input:
- original openLCA JSON-LD database ZIP;
- optional LCIA methods JSON-LD ZIP or folder;
- LCIA indicator selection;
- audit tau values, defaulting to 0.95 and 0.99.

Default output:
- `lcia_flow_priority.csv`;
- `lcia_flow_priority_metadata.json`.

Optional output, not generated by default:
- `lcia_flow_loss_ledger.parquet`.

The optional loss ledger must be controlled by an explicit GUI option or CLI flag.

Default CLI flag should be false.

Suggested CLI flag:
- `--write-loss-ledger`.

## LCIA-critical flow priority CSV

The default priority CSV must contain one row per elementary flow.

Rows are aggregated over all occurrences of that flow across all audited processes.

For default audit tau values 0.95 and 0.99, the CSV columns must be exactly:

- `flow_id`
- `flow_name`
- `compartment`
- `subcompartment`
- `reference_unit`
- `occurrence_count`
- `characterised_occurrence_count`
- `tau_entry_min`
- `tau_entry_median`
- `tau_entry_max`
- `eta_0_95`
- `eta_0_95_witness`
- `loss_max_0_95`
- `eta_0_99`
- `eta_0_99_witness`
- `loss_max_0_99`
- `cf_status`

Do not include `protected` in the compact priority CSV.

Do not include `breaks_0_95` or `breaks_0_99`; these are redundant because certificate breaking is determined by whether eta is positive.

`eta_*_witness` may be included to identify the process, category, and sign that produced the maximum reported shortfall.

## Priority-file mathematical definitions

For each process `i`, selected LCIA category `j`, and elementary exchange occurrence `e` in that process:

```text
a[j, i, e] = exchange_amount(i, e) × characterisation_factor[j, flow(e)]
```

As in the lite-database workflow, use characterised contributions and split signs before running greedy selection logic.

For a fixed process and sign, the greedy ladder induces nested selected prefixes as tau increases.

For a single flow occurrence already present in the retained prefix at audit tau:

```text
loss(j) = contribution(j, e) / full(j)
eta(j, tau) = max(tau - (coverage_before_loss(j) - loss(j)), 0)
```

For the compact priority CSV:

- `loss_max_tau` is the maximum normalised single-flow loss over the audited witness contexts for that flow;
- `eta_tau` is the maximum single-flow shortfall over the audited witness contexts for that flow;
- `eta_tau` is exact for the recorded single-flow witness context, but not an exact database-wide grouped-failure result for dropping that flow everywhere.

For a selected flow set `F`, do not claim exact combined `eta_F(tau)` from the compact CSV alone.

Only claim the conservative screen:

```text
max_{f in F} eta_f(tau) <= exact group eta_F(tau) <= min(tau, sum_{f in F} loss_max_f(tau))
```

If exact grouped failed-set analysis is needed, require a future ledger or recomputation workflow.

## Scope of guarantees

The lite-database tau certificate is local to:
- the selected LCIA categories;
- the resolved characterisation factors and unit conversions used in the run;
- each process after sign-splitting into positive and negative characterised contributions.

Do not claim that the reduction preserves:
- non-selected LCIA methods;
- every possible product-system result;
- non-LCIA uses of elementary exchanges;
- results under different CF-resolution assumptions.
